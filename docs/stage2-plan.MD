# Stage2 实施计划（基于 docs/stage2.MD + docs/navigation.MD + docs/OCR.MD）

## 1. 当前基线与差距

### 1.1 已完成（可复用）
- 数据持久化：localStorage（members + nextId）。
- 成员管理：新增/编辑/递归删除（含循环依赖校验）。
- 树图展示：ECharts 正交树，支持缩放拖拽。
- 导入导出：JSON 导入/导出、覆盖确认、错误提示。
- 搜索基础：按姓名/ID 搜索，结果计数，树图高亮与自动定位（V1.1 完成）。

### 1.2 关键差距
- 关系模型仅支持父子链，尚不支持配偶关系。
- 成员档案字段偏少（无生卒信息、生平、多媒体）。
- 无 OCR 数据补录能力（纸质族谱照片/扫描件无法自动转结构化成员数据）。
- 无 OCR 结果核对工作台与去重合并流程（识别结果难以安全落库）。
- 无打印导出（PDF/图片/Excel）与统计报表。
- 无扫墓轨迹能力（GPX 上传、地图展示、导航跳转、轨迹存储）。
- 无分支管理、事件提醒、协作权限等高级能力。

---

## 2. 迭代原则

1. 先做 P0/P1，保证数据模型可扩展。
2. 每次迭代都提供数据迁移策略（旧数据可读）。
3. 维持纯前端可运行，云能力作为可选增强。
4. 优先交付可验证场景，再做体验增强。

## 3. 里程碑拆分

## M1（P0）：关系模型升级与数据安全（2-3 周）

### 范围
- 1 配偶关系管理（P0）
- 6 数据备份与恢复（P0，补云备份预留）

### 任务
1. 数据模型扩展：
- 新增 `spouseIds: number[]`（支持历史多配偶）。
- 增加迁移器：旧 Member 自动补 `spouseIds: []`。

2. 关系约束：
- 禁止自己与自己结为配偶。
- 支持配偶双向一致性写入（A->B 时自动 B->A）。
- 删除成员时清理所有成员中的该配偶引用。

3. UI 与交互：
- 成员表单增加配偶选择。
- 详情页展示配偶列表，可跳转查看。
- 树图新增配偶连接线（虚线/双线）。

4. 数据安全增强：
- 导出文件加入 `schemaVersion`。
- 导入时执行版本校验与迁移。
- 增加“导入预检查报告”（数量、冲突、修复项）。

### 验收
- 任意成员可维护 0..n 个配偶且双向一致。
- 删除成员后不存在悬挂 spouseId。
- 旧 JSON 可导入并自动迁移。

## M1.5（P0）：扫墓轨迹最小闭环（1-2 周，与 M1 并行）

### 范围
- navigation.MD：TR-01~TR-04、MV-01~MV-02、NA-01

### 任务
1. 轨迹模型与存储：
- 新增 `tracks` 与 `nextTrackId`，存储根对象加入 `schemaVersion`。
- 导入旧数据时自动迁移并补齐轨迹字段。

2. GPX 解析：
- 解析 `.gpx` 中 `trkpt(lat/lon/ele/time)`。
- 计算轨迹点数、总距离、起终点；文件大小限制 10MB。

3. 轨迹管理 UI：
- 新增轨迹管理区：上传 GPX、列表展示、重命名、删除。
- 轨迹可关联成员（墓主），支持按成员查看关联数量。

4. 导航能力：
- 提供“导航”按钮，默认使用网页地图跳转到终点坐标。
- 兼容移动端浏览器打开外部地图应用。

### 验收
- 可上传合法 GPX 并生成轨迹记录。
- 轨迹刷新后不丢失，可删除。
- 点击导航可到达终点坐标（移动端/PC 可用）。

## M1.8（P0）：族谱照片 OCR 识别导入闭环（2-3 周，与 M1.5 并行）

### 范围
- OCR.MD：UP-01/UP-02/UP-04、OC-01/OC-02、PA-01/PA-02/PA-04、IM-01/IM-02

### 任务
1. 图片上传与预处理：
- 支持 jpg/png 上传，移动端支持拍照入口。
- 前端压缩（宽度上限 2000px）后再识别。
2. OCR 识别与解析：
- 对接 OCR Provider（优先云 OCR，保留 Tesseract 兜底）。
- 提取姓名、父母、配偶、生卒等字段，映射 TempMember。
3. 人工核对与纠错：
- 结果表格支持增删改行、关系修正。
- 未识别字段保留 rawText 供人工补录。
4. 去重与导入：
- 按“姓名+父亲”或相似度提示重复候选。
- 用户确认后批量写入 members/spouseIds/parentId，并刷新树图。

### 验收
- 可完成单图导入闭环：上传->识别->核对->导入。
- 关键字段（姓名、父亲）提取准确率在清晰印刷体场景 ≥ 80%。
- 导入后关系结构有效，不产生循环父子关系与无效 spouse 引用。

### 边界（本阶段不含）
- 不做表格结构 OCR（OC-03）与多图自动合并。
- 不做导入历史持久化（IM-03）。
- 不做高级图片编辑（裁剪/旋转）首版能力。

## 4. 技术实施清单

1. 模型版本化：
- 在存储根对象加入 `schemaVersion`。
- 建立 `migrations/` 统一迁移入口。

2. 服务层拆分：
- `relationshipService`（父子/配偶一致性）。
- `exportService`（PNG/PDF/CSV）。
- `filterService`（搜索与组合筛选）。
- `trackService`（GPX 解析、距离计算、点抽稀）。
- `navigationBridge`（多端导航 URL 组装）。
- `ocrService`（Provider 封装：云 OCR / Tesseract）。
- `ocrParserService`（文本到 TempMember 的规则解析）。
- `dedupeService`（导入去重与冲突决策）。

3. 组件演进：
- `MemberForm` 分区（基础信息/关系/档案）。
- `FamilyTreeChart` 支持多关系渲染。
- 新增 `ViewSwitcher` 与 `TimelineView`。

4. 质量保障：
- 核心单测：迁移、关系约束、级联删除、筛选结果。
- 回归清单：导入导出兼容、移动端交互、性能（300+ 节点）。

## 6. 风险与对策

1. 风险：关系模型复杂化导致树图布局混乱。
- 对策：配偶连接与父子连接分层渲染，必要时切换 Graph 视图。

2. 风险：历史数据不兼容。
- 对策：强制 schemaVersion + 迁移报告 + 回滚备份。

3. 风险：纯前端存储容量受限。
- 对策：图片压缩、增量导出、可选云备份接口预留。

4. 风险：导出打印跨浏览器差异。
- 对策：先保证 PNG/CSV 稳定，PDF 采用统一渲染方案。

5. 风险：地图底图与导航服务在不同网络环境下稳定性不一致。
- 对策：默认网页导航兜底；后续评估 OSM/高德双通道策略。

6. 风险：GPX 大文件导致前端解析卡顿。
- 对策：先做大小限制与点数抽稀，后续引入 Web Worker。

7. 风险：OCR 准确率受图片质量与字体影响较大。
- 对策：提供人工核对工作台；未识别字段保留原文并强制确认后导入。

8. 风险：第三方 OCR API 涉及隐私与额度成本。
- 对策：默认提示隐私告知；压缩与限流；保留本地 Tesseract 兜底模式。
