# 扫墓轨迹增强功能需求文档（PRD）- 标点照片/视频支持

## 1. 功能概述
在原有扫墓轨迹功能基础上，针对“两步路”等户外轨迹记录工具导出的 GPX 文件，增加**标点（兴趣点）照片/视频**的支持。系统能够解析轨迹中嵌入的标点及其附件（照片、短视频），并在电子族谱地图上以图标形式展示这些标点，点击后可预览多媒体内容，丰富家族扫墓记录。

---

## 2. 用户故事
- **作为** 家族成员  
- **我希望** 将两步路记录的完整扫墓轨迹（包含沿途拍照、录像）上传至族谱  
- **以便** 后代能看到祖先墓地的实景照片、祭拜过程的视频，加深情感连接

- **作为** 家族管理员  
- **我希望** 系统能自动识别轨迹中的标点（POI）并关联照片/视频  
- **以便** 无需手动逐一添加，节省操作时间

---

## 3. 功能需求

### 3.1 标点解析与展示
| ID | 功能 | 描述 |
|----|------|------|
| WP-01 | 解析 GPX 中的标点 | 读取 GPX 文件中的 `<wpt>` 元素，提取名称、描述、经纬度、时间等信息 |
| WP-02 | 支持两步路扩展字段 | 识别两步路导出 GPX 中的扩展标签（如 `<extensions>` 中的 `<link>`、`<photo>`），提取照片/视频 URL（如果嵌入的是本地路径或网络链接） |
| WP-03 | 地图标点渲染 | 在地图轨迹线上叠加标点图标（相机、摄像等），鼠标悬停显示名称，点击弹出媒体预览 |

### 3.2 媒体文件管理
| ID | 功能 | 描述 |
|----|------|------|
| MF-01 | 照片/视频预览 | 支持常见图片格式（jpg/png/gif）和视频格式（mp4/mov），在弹窗或侧边栏中播放 |
| MF-02 | 本地媒体存储 | 若 GPX 中引用的是本地文件路径（如 `/sdcard/...`），提示用户手动上传对应媒体文件；系统提供批量上传界面 |
| MF-03 | 媒体关联 | 允许用户为标点补充或更换照片/视频，并可删除无用的媒体 |

### 3.3 轨迹管理增强
| ID | 功能 | 描述 |
|----|------|------|
| TM-01 | 轨迹合并 | 支持将多次扫墓的轨迹合并显示（同一成员可关联多条轨迹） |
| TM-02 | 标点导出 | 可将标点及媒体信息导出为 JSON 或 KML，便于备份或分享 |

---

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 存储限制 | 单个轨迹关联的所有媒体文件总大小建议不超过 100MB，避免 localStorage 溢出（可使用 IndexedDB 存储媒体） |
| 解析性能 | 支持最大 20MB GPX 文件，标点数量 ≤ 500 个，解析时间 < 3 秒 |
| 多媒体兼容 | 在 PC 和移动端均能正常预览图片和视频，视频使用 HTML5 `<video>` 标签 |
| 离线能力 | 上传的媒体文件需存储在浏览器本地（IndexedDB），离线时仍可查看已加载的媒体 |

## 5. 数据模型扩展

### 5.1 标点对象（Waypoint）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | number | 唯一标识 |
| trackId | number | 所属轨迹 ID |
| name | string | 标点名称 |
| description | string | 文字描述 |
| lat | number | 纬度 |
| lng | number | 经度 |
| time | timestamp | 记录时间（可选） |
| media | array | 关联的媒体对象数组 |

### 5.2 媒体对象（Media）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | number | 唯一标识 |
| waypointId | number | 所属标点 ID |
| type | string | `image` / `video` |
| data | Blob / string | 存储文件数据（Blob）或数据 URL（小文件） |
| thumbnail | string | 缩略图 dataURL（可选） |
| originalName | string | 原始文件名 |

### 5.3 存储方式
- **轨迹数据**仍存储在 `localStorage`（JSON 格式）
- **媒体文件**使用 **IndexedDB** 存储，避免占用 localStorage 配额（通常仅 5-10MB）

## 6. 交互流程

### 6.1 上传带标点的 GPX 文件
1. 用户选择两步路导出的 GPX 文件
2. 系统解析 GPX：
   - 提取轨迹线数据
   - 提取 `<wpt>` 元素，构建标点列表
   - 识别扩展字段中的媒体链接（如 `<link href="photo.jpg">`）
3. 若检测到本地文件路径，提示用户：“检测到标点引用了本地照片，请手动上传这些文件”
4. 用户可通过文件选择器上传对应照片/视频，系统自动匹配文件名
5. 所有媒体存入 IndexedDB，轨迹数据存入 localStorage
6. 刷新地图，展示轨迹线和标点图标

### 6.2 查看标点媒体
1. 在地图上点击标点图标，弹出信息窗口（或侧边栏）
2. 窗口内显示标点名称、描述
3. 若有照片，显示缩略图，点击放大；若有视频，显示播放器控件
4. 支持全屏播放、下载媒体

### 6.3 手动管理标点
1. 在轨迹详情页，列表展示所有标点
2. 支持新增标点（在地图上点击添加）或编辑现有标点
3. 支持上传/删除/替换媒体

## 7. 技术实现建议

### 7.1 GPX 解析增强
- 使用 DOMParser 解析 XML，遍历 `//wpt` 节点
- 读取 `<name>`、`<desc>`、`<time>`，以及两步路扩展命名空间下的 `<link>` 或 `<photo>`
- 两步路 GPX 常见扩展示例：
  ```xml
  <wpt lat="..." lon="...">
    <name>祭祀点</name>
    <desc>供桌位置</desc>
    <extensions>
      <link href="file:///storage/emulated/0/DCIM/IMG_001.jpg"/>
      <photo>IMG_001.jpg</photo>
    </extensions>
  </wpt>
  ```

  若 <link href> 为网络 URL，可直接使用；若为本地路径，需要用户手动上传对应文件

### 7.2 媒体存储方案
- **IndexedDB** 封装为简单的存取库（如 `idb` 库）
- 媒体文件存储为 `Blob`，并为每个媒体生成唯一 ID
- 缩略图生成：图片使用 `canvas` 绘制缩略图（尺寸 100x100），视频需借助 `video` 元素截取第一帧（注意跨域限制）

### 7.3 地图标点图层
- 使用 Leaflet 的 `L.marker` 添加自定义图标（相机、摄像）
- 为每个标点绑定 `popup`，内容动态生成媒体预览

### 7.4 性能优化
- 标点数量多时，使用聚类（Leaflet.markercluster）避免渲染过多
- 媒体懒加载：仅在地图视口内的标点才加载媒体预览

## 8. 界面原型（文字描述）
- **轨迹详情页**：增加“标点”标签页，列表形式展示所有标点及其媒体预览
- **地图页面**：轨迹线旁显示相机图标，点击后弹出卡片，展示照片/视频
- **上传界面**：检测到本地文件引用时，显示文件匹配对话框，引导用户上传
